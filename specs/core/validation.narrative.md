# Validation

## Intent

Валидация markdown после операций записи (create_note, update_note, update_section, append_note).

Цель: предотвратить сохранение сломанного markdown, который приведёт к проблемам рендеринга в Obsidian.

Библиотека: pymarkdownlnt.

Проверки:
- Таблицы: пустая строка перед таблицей
- Code blocks: закрытие ```
- Заголовки внутри code blocks: ложные # (когда code block не закрыт, последующие # парсятся как заголовки)

## Clarifications

Q1: pymarkdown не имеет встроенного правила для "пустая строка перед таблицей" — только MD031 (blanks-around-fences) и MD032 (blanks-around-lists). Варианты:
- a) Написать кастомный плагин для pymarkdown
- b) Реализовать проверку таблиц вручную (regex/парсинг)
- c) Отказаться от проверки таблиц
Какой вариант?
A1: Точно проверяем таблицы. Выбери самое простое решение. Ты же можешь определить положение таблицы? Определить что есть таблица по правилам MD? Ну и проверяй что есть отсуп от текста выше

Q2: pymarkdown не детектирует unclosed code blocks как lint rule — это ошибка парсинга. Варианты:
- a) Отлавливать ошибки парсинга pymarkdown
- b) Реализовать проверку закрытия ``` вручную (подсчёт пар)
- c) Использовать другую библиотеку для этой проверки
Какой вариант?
A2: Вариант b — проверка закрытия вручную (подсчёт пар). Если есть готовое решение — использовать готовое, иначе считать пары.

Q3: "Заголовки внутри code blocks" — это симптом unclosed code block (текст после незакрытого ``` парсится как обычный markdown). Если решим Q2, эта проверка автоматически покрывается. Подтвердите или уточните отдельные требования.
A3: Да, всегда работаем с первопричинами.

Q4: Поведение при ошибке валидации:
- a) Блокировать операцию записи, выбрасывать исключение
- b) Записать, но вернуть warning в результате операции
- c) Записать, логировать warning
Какой вариант?
A4: - b) Записать, но вернуть warning в результате операции

Q5: Валидация применяется ко всему контенту заметки или только к изменяемой части (для append_note, update_section)?
A5: Ко всему

## Extension: English-only naming

Новые требования:
- Note names must be English only (no Cyrillic)
- Heading names (## / ### etc.) must be English only

Q6: Паттерн для "English only" — что именно разрешено?

- a) Только ASCII letters (a-zA-Z) + numbers + spaces + базовая пунктуация
- b) Полный Latin Extended (включая ñ, ü и т.д.)
- c) Строго ASCII printable (0x20-0x7E)

Какой вариант?
A6: Вариант А + смайлики Obsidian Tasks: ➕⏳🛫📅✅❌⏬🔽🔼⏫🔺🔁🏁🆔⛔

Q7: Когда валидировать note names?

- a) При create() — блокировать создание с кириллицей
- b) При create() — создать, но вернуть warning
- c) Отдельный метод validate_name() без интеграции в create

Какой вариант?
A7: Вариант А — блокировать при create() и rename()

Q8: Headings — проверять при той же валидации контента что и tables/code blocks?
A8: Да, при общей валидации контента

## Extension: Wikilinks section validation + heading blocking

Новые требования:
- `[[Note#Section]]` — секция должна существовать в целевой заметке
- Заголовки должны быть English-only (blocking, как для имён)

Q9: Сейчас cyrillic-heading — это warning. Пользователь хочет blocking error как для имён?
- a) Blocking error (как validate_name) — запись не происходит
- b) Оставить warning (запись происходит, возвращается предупреждение)
A9: Давай Blocking error, а то ты часто игноришь предупреждения

Q10: Wikilinks `[[Note#Section]]` — когда секция не существует:
- a) Warning (post-write, запись происходит)
- b) Blocking error (запись не происходит)
A10: Blocking error

Q11: Для проверки секции нужен доступ к Vault (чтобы прочитать целевую заметку). Validator сейчас stateless. Варианты:

**Проблема**: `[[Note#Section]]` → нужно открыть `Note.md` и проверить есть ли заголовок `# Section`. Validator сейчас получает только `content: str`, без доступа к файлам.

**a) Передавать Vault в validate(content, vault)**
- (+) Единая точка валидации
- (+) Простой API: один метод validate() проверяет всё
- (-) Validator становится зависим от Vault (coupling)
- (-) validate() меняет сигнатуру — ломает существующий код

**b) Отдельный метод validate_wikilinks(content, vault)**
- (+) Validator остаётся stateless для остальных проверок
- (+) Явное разделение: validate() — markdown, validate_wikilinks() — ссылки
- (-) Два вызова вместо одного в Operations
- (-) Валидация разбита на части

**c) Проверку делать в Operations, не в Validator**
- (+) Operations уже имеет Vault
- (+) Validator остаётся чистым (только markdown)
- (-) Логика валидации размазана по двум местам
- (-) Дублирование: каждая операция (create, update, append) должна вызывать проверку

**Рекомендация**: Вариант (b) — баланс между чистотой и удобством.

A11: Делаем в Operations. Алгоритм:
1. vault.extract_wikilinks(content) → список ссылок
2. Парсим `[[Note#Section]]` → Note + Section
3. Проверяем Note существует (vault.get_note)
4. Если есть Section → get_headings(Note) и ищем heading
5. Нет → raise BrokenLinkError

Q12: Битые wikilinks `[[NonExistentNote]]` (без секции) — тоже проверять?
A12: Warning (forward reference допустим). Blocking только для `[[ExistingNote#NonExistentSection]]`.

Status: resolved

## Extension: Section size validation

Новые требования:

- Автоматическая проверка размера h2-секций (topic level)
- Порог: 5000 символов
- Если превышен → добавить `[EXTRACT]` в заголовок как подсказку для рефакторинга
- Проверять только h2, не h3/h4
- Warning, не blocking error

Q13: Что считаем "размером секции"?

- a) От ## до следующего ## (или конца файла)
- b) От ## до любого следующего заголовка (#, ##, ###)

A13: Вариант (a) — от ## до следующего ## или конца файла

Q14: Включать ли подзаголовки (###, ####) в подсчёт размера?
A14: Да, включать весь контент до следующего ##

Q15: Добавлять `[EXTRACT]` автоматически в заголовок или только возвращать warning?
A15: Только warning с рекомендацией. Не модифицировать контент автоматически.
